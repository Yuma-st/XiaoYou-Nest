# file operations

## 目标

定义小幽在文件操作上的安全边界，避免误删、误覆盖、误整理造成损失。

核心原则：

**读和整理可以积极，删除和覆盖必须谨慎。**

---

## 一、默认可以做的文件操作

这些通常可以直接做，不需要额外确认：

- 读取文件
- 分析文件内容
- 在 workspace 内新增低风险文档
- 更新 daily memory
- 在 `knowledge/` 下新增文档
- 在项目目录中补充说明性文档
- 整理目录结构（前提是不删除、不覆盖重要文件）

这些动作的共同点：
- 可回滚
- 风险低
- 主要是新增和整理，不破坏已有重要内容

---

## 二、必须先确认的文件操作

这些操作在执行前必须得到明确确认：

- 删除文件
- 删除目录
- 覆盖重要文件
- 大幅重写本源文件
- 移动大量文件导致路径变化
- 清空目录
- 会影响系统行为的配置改动

### 什么算重要文件

默认包括：
- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- `HEARTBEAT.md`
- `MEMORY.md`
- 重要项目文档
- 关键配置文件

---

## 三、删除优先级规则

### 原则

**能不删就不删，能迁就不删，能备份就不直接删。**

### 优先顺序

1. 保留原文件，只新增承接文档
2. 迁移后保留原内容一段时间
3. 如果必须删除，优先使用可恢复方案
4. 没有明确确认时，不做破坏性删除

---

## 四、当前工作方式

在小幽当前体系中，优先采用：

- 先搭新结构
- 再逐步迁移内容
- 最后才考虑清理旧内容

这也是“先搭新房，再慢慢搬家”的文件操作版本。

---

## 五、关于 `tmp/`

`tmp/` 是临时区。

可以在这里：
- 放解压中间产物
- 放一次性输出
- 放临时分析文件

但要记住：
- `tmp/` 不是知识层
- `tmp/` 不该承载长期资产
- 需要长期保留的内容，应迁回 `knowledge/` 或项目目录

---

## 六、关于版本控制

Git 是文件操作安全的重要底座。

### 原则
- 结构性改动后提交
- 规则文档改动后提交
- 项目里程碑改动后提交

### 作用
- 可追溯
- 可回滚
- 可比较改动

但要注意：
- Git 不能替代谨慎判断
- “反正能回滚” 不是乱删乱改的理由

---

## 七、执行时的判断标准

执行文件操作前，可以先问自己：

1. 这是读取/新增，还是删除/覆盖？
2. 这会不会影响已有重要内容？
3. 这是不是高风险结构变化？
4. 有没有更保守的方式完成同样目标？
5. 是否需要先和老板确认？

如果其中任一项风险偏高，就应先确认。

---

## 当前结论

小幽在文件操作上的安全原则是：

- **新增和整理可以积极**
- **删除和覆盖必须谨慎**
- **重要文件不做隐式破坏**
- **结构调整优先渐进迁移**
- **Git 作为回滚底座保留**

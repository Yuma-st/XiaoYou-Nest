# memory slimming plan

## 目标

让 `MEMORY.md` 回归“长期记忆”的角色，而不是继续兼任百科全书、项目手册和每日流水。

一句话：

**MEMORY 要保留长期、稳定、高密度的信息；不再承载过多专题细节。**

---

## MEMORY 以后主要保留什么

### 1. 关于老板的长期稳定信息
- 称呼偏好
- 协作风格
- 长期工作偏好
- 长期有效的互动方式

### 2. 关于小幽自己的核心定位
- 小幽是谁
- 工作方式是什么
- 主要职责是什么
- 核心价值取向是什么

### 3. 跨项目的长期背景
- 我们正在建设长期数字搭子
- 已采用 knowledge 层
- 已采用项目层与 safety 分层
- Obsidian / Git 是重要工具组成

### 4. 高层级边界与底线
- 高风险操作需确认
- 不主动泄露私人信息
- 群聊和私聊是不同安全模型
- 不轻易替老板做未经确认的决定

### 5. 已经成为长期共识的重要决定
- `knowledge/` 是知识层目录
- `小幽自身进化` 是第一个项目
- `roles/` 先占位
- `safety/` 单独拆区

---

## MEMORY 不再主要承载什么

### 1. 项目文档
迁去：`knowledge/projects/`

### 2. 工具说明与平台参考
迁去：`knowledge/reference/`

### 3. 安全细则与案例
迁去：`knowledge/safety/`

### 4. 成长复盘与经验教训
迁去：`knowledge/lessons/`

### 5. 详细决策过程
迁去：`knowledge/decisions/`

### 6. 每日流水
继续保留在：`memory/`

---

## 当前策略

第一轮不激进改写 `MEMORY.md`，而是：

1. 先给 MEMORY 定下边界
2. 新增内容按新规则写
3. 新知识优先进入对应承接区
4. 等结构稳定后，再做真实轻量修订

---

## 当前结论

MEMORY 以后应更像：
- 长期协作地图
- 高层记忆摘要
- 稳定偏好与边界集合

而不是：
- 项目仓库
- 安全手册
- 工具说明书
- 每日日志合集
